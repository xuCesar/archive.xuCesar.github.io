一次i18n的记录

发表于 2018-10-04 | 分类于 前端技术 | 阅读次数

i18n——Internationalization的简称，它表明的意思是使产品具有能轻松适配指定语言和文化的能力。听起来相当唬人啊！

项目背景

为了满足客户日益增长的各种需求，不得不摆好各种姿势~

项目介绍

此次i18n是基于已在生产环境运行的项目（暂且称之为S）进行的，并在此基础上对ui进行了较大的改动。项目的核心功能是赛事报名，主要分为个人报名、组团报名、接力赛以及优惠码，具体的业务流程这里就不分析了…（主要是比较复杂，我自己也比较懵逼）

S项目前端使用了React栈，采用react-router进行路由管理，采用redux进行数据和状态管理，采用antd-mobile作为前端ui框架，采用axios对http请求进行管理，以及其他各类类库。

项目历史

S项目起初由咪总和阿总完成的，然后交接给了华总，此次本地化由给我和约总完成。

项目需求

S项目面向的主要用户是中国用户，存在少量的外籍用户。所以项目方希望在中文版本的基础上，实现一个英文版本，根据用户设备选择语言，记录上一次语言选择，并能够切换语言。

这个需求的很多细节是值得推敲的，诸如以下：

  • 产品始终在中国运行，货币符号始终采用¥
  • 时间日期格式并不完全遵循西方习惯
  • 数字格式并不完全遵循西方习惯
  • 既定的语言文本
  • …

所以，并不能算是一个标准的i18n项目。

着手i18n

S项目是基于React栈的，所以也没有怎么思考，就直接选用了Yahoo出的国际化组件react-intl，它提供了用于格式化日期、数字和字符串的api、复数和翻译处理。实际上在项目里，只用到了一些api和组件。

使用LocaleProvider组件，并赋值。

同时，我们需要使用react-intl提供的IntlProvider组件，来引入自定义文案，IntlProvider组件和LocaleProvider组件类似，用于设置i18n的上下文。主要接受3个属性：

  • loacle 当前的语言环境
  • messages 当前语言的内容
  • key 当前的语言环境

这里要强调的是，key这个属性。前面我们说了一个需求：动态的切换语言。这个问题困扰了我很久，最后在T大佬的帮助下，找到了答案。引一段react-intl文档上的一段话：

Dynamic Language Selection
By default, changes to the locale at runtime may not trigger a re-render of child elements. To solve this, and enable dynamic locale modification, add a key property to the and set it to the locale, which persuades React that the component has been modified:

大致意思是说，在程序运行时更新语言环境，不会触发页面的重新渲染。想要动态修改语言环境并重新渲染，必须通过key属性的变化来通知IntlProvider组件，语言环境已经发生改变，需要重新渲染页面。所以，这个属性是至关重要的。

使用高阶组件（HOC）injectIntl，将传入的React组件与另一个React组件包装在一起，并通过props将格式化的api注入到包装组件中。在组件内部可以使用this.props.intl拿到注入的api和组件。
