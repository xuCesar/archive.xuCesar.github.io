Git 学习笔记（二）

发表于 2018-08-11 | 分类于 前端技术 | 阅读次数

上一篇主要说了git的安装以及基本操作，这篇继续研究git的分支。

分支

当我们初始化仓库的时候，系统会默认创建一个名为master的分支，其也被约定俗成为版本库的主分支。

继续上一个test仓库。查看下当前本地版本库的内容以及状态。当前提交的分支为origin下的master分支。为什么要查看下版本库的状态呢？

  • 跟踪文件的状态
  • 确保当前设备在当前分支下是否存在未提交的更改，减少冲突的产生。

所以，随时查看状态是一个好习惯~~🤔

首先git branch查看下分支，*表示当前版本在哪个分支下。

创建分支

使用git branch dev创建一个分支名为dev的新分支，分支名随意~

切换分支

使用git checkout dev切换到刚才创建的dev分支，注意*已经在dev的前面了

我们在dev分支上新建一个index.js文件，然后查看状态并push到远端。

ok，我们对比一下master和dev分支下的内容。可以看到，master分支下少了一个刚才创建的index.js文件。这也印证了前面说的的分支之间各有状态，互相独立。

分支合并

初始化的时候会创建master分支，默认为主分支。当我们在dev分支上的开发完成后，使用git merge将dev分支上的内容合并到master主分支。下图可以看到index.js文件已经在master分支上了。所以，dev已经被成功合并。

解决冲突

使用git checkout -b examination创建并切换到examination分支。修改index.js文件。执行git commit -a -m命令将修改提交到分支。

切换到master分支，并查看状态。

继续修改index.js文件，新增一个函数并提交。这个时候执行git merge就会引发冲突。

这个时候是可以git push，但是冲突还会存在。所以我们还是要把冲突解决掉。查看工作区的文件，会发现冲突的地方已经被打上了tag，我们可以很清晰的处理掉冲突。

我们可以手动解决冲突，也可以借助IDE的插件快速解决，看你习惯了~解决完冲突，git add . git commit -m，然后git status查看状态，可以看到master超前2个版本~

ok,执行git push推送~

拉取版本

继续切换到examination分支git checkout examination，执行git pull origin master，拉取master分手上更新的内容到examination分支。

命令总结

git branch dev：切换分支

git branch -d dev: 删除分支

git checkout dev：创建分支

git checkout -b examination： 创建并切换分支

git merge: 合并分支

git pull origin master：拉取远端分支版本

结语
